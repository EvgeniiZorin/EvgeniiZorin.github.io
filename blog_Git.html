What is Git and how to use it - a personal insight

In my journey of learning programming, I have come a long way in implementing version control. At first, my idea of version control was to pack the current state of a program in a folder, call it with date and version number, and then duplicate the folder if i wanted to continue working into a new version. I have heard of Git at that time, but I didn't entrust my version control to a system that I cannot see. Currently, I cannot imagine my life without using Git for versioning my programs! In this blog entry, I would like to write about why I love Git so much.

There are a number of reasons I love using Git:

  1. Once you learn the basic commands, It is very easy to switch between versions of your code, if you decide to change things around;
  2. I store my programs on Github, and with Git it is very easy to download the current (or a specified) version of any program to my local computer, edit, then push the new version to Github!

What is git

In short, Git is a type of version control system (VCS) that keeps track of diffferent changes made to your code. In essence, you can run Git in CLI to save states of your programs in different times of editing, just like you can do in a computer game by saving your progress. Git allows the programmer to revert to any state (or version) of the program.

Basic commands

First, we need to install Git locally on your PC. You can check Git version by git --version and check current configurations, such as user.name and user.email, by git config -l.

You could create a repository and files locally, start tracking from local computer, and then push to github by the following steps:

  • Create a local repo, make changes
  • git init
  • git add .
  • git commit -m "Message"
  • Create a repo on Github, copy the HTTPS link
  • git remote add origin [https]
  • git push -u origin main

    • To clone a repository from Github, you can use the command git clone [ssh / https address]. Cloning is bringing a repository hosted on Github to your local PC. You can use SSH or HTTPS, depending on which one works for you. If you already have a repository from Github, but the one on Github has changes that weren't updated on the local PC, we can update local repo from the latest changes in Github with git pull --rebase or git pull origin main, instead of "main" you could try "master". Alternatively, if you would like to start a new project and start tracking it from your local machine, you can use git init.
    Once you have started or cloned Git repository, you can check the status of your work - git status - which will tell you which files have been changed.
    To check commit history, use git log:

  • The tag --oneline will show one version per line, which is very convenient
  • -1 will only show the last commit
  • git diff will show differences and alterations

    • When we have made the changes and would like to save the current state of the program, we first need to add changes to staging/tracking:
  • git add . to add all files changes
  • git add filename.ext to track just one file

    • Subsequently, we need to commit changes, meaning that the changes will be added as a version to Git system. This can be done by git commit -m "Comment here". If you made a recent commit, but forgot to make small edits in that commit, you can recommit the latest change to add it onto the latest commit:
  • git add .
  • git commit --amend
  • git push --force

    • Finally, we can push the changes made online to hosting repository: git push.

    Working with branches

    I rarely use branches in my Git version control. This feature can be useful to create different branches from a chosen version, to perhaps explore different ways to edit the code.

  • Delete a branch git branch -d branchName
  • Create a branch git branch branchName
  • Switch to a different branch git checkout branchName
  • Create and switch to a new branch git checkout -b branchName
  • Merge a chosen branch with the working (current) branch git merge branchName
